HAYSTACK

Como simular AirTags no seu próprio hardware e receber as localizações no seu servidor, usando a rede de centenas de milhões de iPhones no mundo.

⚠️ Esta arquitetura usa o protocolo Find My da Apple de forma não oficial. É um projeto acadêmico open-source (TU Darmstadt). A Apple pode alterar o protocolo em atualizações do iOS — considere isso um risco operacional antes de colocar em produção.
01 · O fluxo completo — do beacon ao seu mapa
📡
Beacon nRF52832 na moto Hardware
Transmite BLE advertising continuamente. O pacote imita exatamente o formato de um AirTag: contém a chave pública EC efêmera do beacon. iPhones nas redondezas reconhecem como dispositivo Find My legítimo.
// Pacote BLE transmitido pelo beacon
Tipo: BLE Advertisement (não-connectable)
Conteúdo: Public Key (28 bytes, curva P-224)
Intervalo: 200ms
Alcance: ~10–30 metros
Bateria: meses (CR2032 ou bateria da moto)
📱
Qualquer iPhone nas redondezas Automático · sem app
O iOS detecta o pacote BLE, pega sua própria localização GPS e cifra a localização com a chave pública do beacon. Envia o pacote cifrado para os servidores da Apple. O dono do iPhone não vê nada, não autoriza nada — é automático.
// O que o iPhone faz internamente
1. Detecta beacon BLE → extrai Public Key
2. Pega GPS próprio: lat=-8.28, lng=-35.97
3. Cifra: ECIES(PublicKey, {lat, lng, ts})
4. POST → apple-findmy-servers
// Ninguém lê exceto quem tem a Private Key
☁️
Servidores da Apple (Find My) Apple Infrastructure
A Apple armazena os pacotes cifrados associados ao hash da chave pública. Ela não consegue descriptografar — só funciona como um repositório seguro. Os dados ficam disponíveis para consulta por ~24h.
Endpoint: gateway.icloud.com/acsnservice/find
Chave de busca: SHA256(PublicKey)
Retorno: [ {datePublished, payload_cifrado} ]
Retenção: ~24 horas por registro
🐍
Script Python no seu servidor Seu código
Um script Python roda a cada poucos minutos, consulta a API da Apple com o hash da chave pública de cada beacon, baixa os pacotes cifrados e os descriptografa com sua chave privada. A localização real aparece só aqui.
# Loop de coleta (roda a cada 5 min)
for beacon in frota:
  reports = fetch_apple(beacon.pubkey_hash)
  for r in reports:
    lat, lng = decrypt(r, beacon.privkey)
    salvar_no_banco(beacon.id, lat, lng, r.ts)
🗺️
Dashboard da Autovip Seu servidor
As coordenadas descriptografadas ficam no seu PostgreSQL. O painel web mostra a última posição conhecida de cada veículo no mapa. O associado acessa e vê a moto dele. Você controla tudo — nenhum dado passa pela Apple em aberto.
Stack: FastAPI + PostgreSQL + Leaflet.js
Atualização: a cada 5 minutos (polling Apple)
Multi-tenant: cada associado vê só seus veículos
Histórico: armazenado no seu banco (ilimitado)

03 · Os 4 componentes que você precisa construir
📡
Firmware do Beacon
Firmware open-source do OpenHaystack para nRF52832 ou ESP32. Você flasha com a chave pública gerada para aquele veículo específico. Cada beacon tem chaves únicas.
nRF52832 OpenHaystack firmware ~R$15/unidade
🐍
Script de Coleta Python
Script que autentica com Apple ID (dedicado), consulta a API Find My a cada 5 minutos para cada beacon, descriptografa os pacotes e salva no banco.
Python 3 findmy library Apple ID dedicado
🗄️
Backend + Banco de Dados
FastAPI expõe endpoints REST para o dashboard. PostgreSQL guarda o histórico de posições. Tabela positions com veiculo_id, lat, lng, ts, associado_id.
FastAPI PostgreSQL Docker
🗺️
Dashboard SaaS
Interface web com Leaflet.js mostrando a última posição de cada veículo do associado logado. Histórico de percursos, alertas e geofence.
Leaflet.js OpenStreetMap Supabase Auth
04 · Passo a passo para implementar
1
Clonar o OpenHaystack e gerar os pares de chaves
Repositório: github.com/seemoo-lab/openhaystack — Usar o script Python do projeto para gerar um par de chaves EC (P-224) para cada beacon. Guardar as chaves privadas com segurança no seu servidor. Nunca expor.
Dia 1 GitHub
2
Gravar o firmware no nRF52832 com a chave pública
O OpenHaystack tem firmware pronto para nRF52832. Você compila injetando a chave pública de cada beacon e grava com nRF Connect Desktop ou J-Link. Um beacon por veículo.
Dias 2–4 Firmware
3
Criar um Apple ID dedicado para consultas
Criar um Apple ID separado apenas para este sistema. Não usar conta pessoal. Ativar autenticação de 2 fatores. A biblioteca Python findmy usa esse Apple ID para autenticar nas consultas à API Find My.
Dia 2 Apple ID
4
Subir o script Python de coleta no servidor
Usar a biblioteca open-source "findmy" (PyPI) ou o script do OpenHaystack para consultar a API Apple a cada 5 minutos. Para cada beacon: busca relatórios → descriptografa com chave privada → extrai lat/lng → salva no PostgreSQL.
Dias 3–5 Python
5
Construir o backend FastAPI + PostgreSQL
API REST com endpoints: GET /veiculos/{id}/posicao (última posição), GET /veiculos/{id}/historico (últimas 24h). Multi-tenant: cada associado só acessa os próprios veículos via JWT.
Semana 2 Backend
6
Dashboard web com mapa Leaflet
Interface SaaS com login dos associados (Supabase Auth). Mapa com marcadores dos veículos, última atualização, histórico de percurso e alerta quando veículo sair de área configurada (geofence).
Semanas 3–4 Frontend
05 · Riscos que você precisa aceitar
🍎
Apple pode mudar o protocolo a qualquer momento
Uma atualização do iOS pode quebrar a compatibilidade sem aviso. Aconteceu antes com versões do OpenHaystack. Monitorar o repositório GitHub do projeto é obrigatório.
ALTO
📵
Só funciona com iPhones — Android não participa
A rede Find My da Apple só é alimentada por dispositivos iOS. Em regiões com poucos iPhones a cobertura pode ser baixa. No Brasil iPhones representam ~30% dos smartphones.
MÉDIO
⏱️
Latência de 5–15 minutos, não é tempo real
O ciclo é: beacon transmite → iPhone coleta e envia → Apple armazena → seu script busca e decifra. O processo todo leva entre 5 e 15 minutos. Para localização pós-furto é suficiente. Para rastreamento ao vivo, não.
